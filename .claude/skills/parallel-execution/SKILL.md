# 部门内并行执行

> 六部收到 ≥ 2 个子任务时，切换为堂官-员外郎模式并行执行。

## 触发条件

- 尚书省同时向同一部门派发 ≥ 2 个子任务
- 或部门已有 1 个进行中任务，又收到新的无冲突任务
- 仅 1 个子任务时**不启动**堂官模式，直接执行

## 角色

**堂官**（部门主 Teammate）：冲突检测 → 任务分组 → 申请创建员外郎 Teammate → 汇总产出 → 回报尚书省。堂官模式下原则上不直接产出，仅协调。例外：串行链仅含 1 个任务时堂官可自行执行。

**员外郎**（Teammate Worker）：堂官申请创建的独立 Teammate，继承部门核心身份，在独立 worktree 中执行单个子任务，完成后通过 SendMessage 向堂官回报。

## 上级并行提示（parallel_hint）

堂官在冲突检测前，须先查看工单中尚书省附加的 `parallel_hint`：

- `confidence: high` + 堂官验证无冲突 → 直接并行
- `confidence: medium` + 堂官验证无冲突 → 并行
- `confidence: medium` + 堂官发现冲突 → 改为串行，回报尚书省说明原因
- `confidence: low` 或无 hint → 堂官自行完整检测后决定

**hint 是建议不是命令**：堂官有权基于实际冲突检测推翻上级标注，但推翻时须在回报中说明原因。即使上级未附加 hint，堂官仍须自行判断是否可并行。

## 冲突检测

```
1. 参考上级 parallel_hint（若有）
2. 提取每个任务的影响范围（文件路径、模块、外部资源）
3. 两两比较影响范围
   - 无交集 → 可并行
   - 有交集 → 必须串行
4. 输出：并行组 + 串行链
```

### 各部门判定细则

| 部门 | 可并行 | 必须串行 |
|------|--------|---------|
| 🔧 工部 | 不同文件/模块 | 同一文件、有函数调用依赖 |
| 📝 礼部 | 不同文档文件 | 同一文档、有交叉引用需同步 |
| ⚖️ 刑部 | 不同模块的测试/审查 | 集成测试（需全部代码就绪） |
| ⚔️ 兵部 | 不同服务/环境 | 同一服务的配置变更 |
| 💰 户部 | 不同数据表/数据源 | 同一表的 schema 变更与数据迁移 |
| 📋 吏部 | 不同 Agent 的配置 | 全局权限规则变更 |

## 执行流程

```
收到 N 个子任务（N ≥ 2）
  │
  ├── 堂官提取影响范围，划分 [并行组] + [串行链]
  │
  ├── 并行组：
  │     ├── 每个任务申请创建 1 个员外郎 Teammate（独立 worktree）
  │     ├── 员外郎并行执行，通过 SendMessage 向堂官汇报
  │     └── 全部完成 → 堂官收集产出
  │
  ├── 串行链：
  │     ├── 仅 1 个任务 → 堂官自行执行
  │     └── ≥ 2 个任务 → 逐个启动员外郎按序执行
  │
  └── 堂官汇总
        ├── 检查产出间是否有 merge conflict
        ├── 有冲突 → 堂官协调解决或上报尚书省
        └── 无冲突 → 合并产出，统一回报尚书省
```

## 约束

1. **员外郎上限**：单部门同时不超过 **3 个**，超出排队
2. **必须隔离**：员外郎使用独立 worktree，禁止同目录并行修改
3. **员外郎不互通**：所有交互经堂官中转
4. **失败处理**：员外郎失败时堂官按容错规则处理（重试/替换/上报）
5. **创建审批**：员外郎 Teammate 的创建须经尚书省确认
6. **数量上限**：单部门不超过 3 个，全局不超过 12 个
7. **刑部审查**：堂官合并员外郎产出后，整体产出经正常流程送刑部审查。刑部驳回时，由尚书省退回堂官，堂官按需重新启动对应员外郎修复
