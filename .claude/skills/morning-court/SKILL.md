# 早朝开始

> 皇上驾到，早朝开始。太子主持，逐级收集，最终呈上《早朝奏报》。

## 执行流程

### 第零阶段：团队就绪检查

太子检查当前会话是否已创建 Team 与三省 Teammate：
- 若已创建 → 直接进入**第一阶段**
- 若未创建 → 加载 `team-bootstrap` Skill，完成 TeamCreate 与三省 Teammate 创建后继续
- 若为首次会话且无历史任务 → 可跳过第二/三阶段，仅执行第一阶段自查 + 第四阶段朝报

### 第一阶段：太子自查

太子独立收集以下信息，**不唤醒任何 Teammate**：

1. **读取起居注**：检查 `.claude/qijuzhu/` 目录，读取最近记录，了解上次会话的执行脉络与遗留事项
   > 📎 起居注格式与写入规则见 `qijuzhu-format.md`
2. **读取持久化状态**：通过 TaskList 检查当前所有任务状态
3. **检查工程最新状态**：`git log`（最近提交）、`git status`（未提交变更）、`git branch`（当前分支）
4. **扫描阻塞与待决**：是否有 `blocked` 任务、是否有三驳上奏待裁决、是否有覆奏待审

**太子判定**：
- 若信息已充分（所有任务状态清晰、无异常） → 直接进入**第四阶段**编写朝报
- 若存在模糊状态、需要决策层补充 → 进入**第二阶段**

### 第二阶段：唤醒三省

太子通过 SendMessage 联络需要的三省 Teammate，**按需而非全部**：

| 条件 | 唤醒对象 | 询问内容 |
|------|---------|---------|
| 有草案待审议或封驳中 | 中书省、门下省 | 当前方案状态、封驳轮次、修订进展 |
| 有任务执行中或协调中 | 尚书省 | 各部执行进度、阻塞原因、预计完成时间 |
| 有覆奏待审 | 门下省 | 覆奏审查进展、发现的问题 |

**三省判定**：
- 若三省自身信息足以回答 → 直接返回状态给太子，进入**第四阶段**
- 若需要六部提供具体执行细节 → 进入**第三阶段**

### 第三阶段：三省唤醒六部

由尚书省决定唤醒哪些六部，**仅唤醒有活跃任务的部门**：

| 条件 | 唤醒对象 | 询问内容 |
|------|---------|---------|
| 有开发任务执行中 | 工部 | 代码进度、技术障碍 |
| 有测试/审查进行中 | 刑部 | 审查结果、未通过项 |
| 有数据任务执行中 | 户部 | 数据处理进度、资源消耗 |
| 有部署/运维任务 | 兵部 | 部署状态、监控告警 |
| 有文档任务执行中 | 礼部 | 文档完成度 |
| 有 Agent 管理任务 | 吏部 | Teammate 健康状态、权限变更 |

无活跃任务的部门**不唤醒**。

### 第四阶段：太子呈上朝报

太子整合所有收集到的信息，以固定格式编写《早朝奏报》呈给皇上：

```
📜 早朝奏报

一、系统状态
    各 Teammate 当前状态（在线/离线/熔断）

二、进行中旨意
    列出所有 active plan_id、当前阶段、负责 Teammate

三、阻塞项
    blocked 任务清单，含阻塞原因与建议处置

四、已完成待御览
    已交付但皇上尚未查阅的产出物

五、待皇上裁决
    需要皇上决策的事项（三驳上奏、降级审批等）

六、工程概况
    最近 git 提交摘要、当前分支、未提交变更
```

## 朝报原则

1. **简明扼要**：每一项用一句话概括，皇上一眼看清全局
2. **无事不扰**：某项无内容时写"无"，不展开空话
3. **有事必报**：阻塞项和待裁决事项必须列出，不得隐瞒
4. **太子口吻**：朝报由太子以恭谨但不失灵动的风格呈上
